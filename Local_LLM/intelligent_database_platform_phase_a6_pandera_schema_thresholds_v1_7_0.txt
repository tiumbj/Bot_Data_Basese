============================================================

- timeframe: string

- open: float64
- high: float64
- low: float64

- tick_volume: int64
- spread_points: int32 nullable
- real_volume: int64 nullable
- ingest_run_id: string

- logical_key: string
- row_hash: string

- quality_status: string
- quality_score: float64
- continuity_status: string
- conflict_status: string nullable
- anomaly_flags: list[string] nullable

Field meanings:
- logical_key = symbol|venue|timeframe|ts_utc
- row_hash = hash ของ row payload ที่ normalized แล้ว
- batch_fingerprint = hash ของ raw batch
- quality_status = PASS/WARN/FAIL/QUARANTINE
- continuity_status = OK/GAP/DUPLICATE/REPAIRED/CONFLICT
- conflict_status = NONE/HASH_MISMATCH/OHLCV_MISMATCH/LATE_REPLACEMENT

5) NORMALIZATION RULES BEFORE VALIDATION
============================================================
5.1 Time
- ทุก timestamp ต้องแปลงเป็น UTC ก่อน
- ห้ามเก็บ naive datetime ใน canonical layer
- canonical output เก็บ ts_utc เท่านั้น

5.2 Symbol and timeframe
- symbol uppercase
- timeframe normalized เป็นหนึ่งใน:
  M1 M2 M3 M4 M5 M6 M10 M15 M30 H1 H4 D1
- ingest row-level raw bars เริ่มที่ M1 เท่านั้น
- TF อื่นเป็น derived deterministic products

5.3 Numeric normalization
- open/high/low/close cast เป็น float64
- tick_volume cast เป็น int64
- spread_points cast เป็น int32 nullable
- real_volume cast เป็น int64 nullable

5.4 Sorting
- sort by symbol, venue, timeframe, ts_utc, ingest_ts_utc

5.5 Null policy
- open/high/low/close/tick_volume/ts_utc/symbol/timeframe null = FAIL
- spread_points/real_volume nullable ได้ แต่ต้องถูก normalize

6) PANDERA CONTRACT SPEC
============================================================
6.1 Table-level contract
Contract name:
mt5_canonical_ohlcv_m1_contract

Severity levels:
- PASS
- WARN
- FAIL
- QUARANTINE

Decision rule:
- FAIL และ QUARANTINE ไม่เข้า canonical DB
- WARN เข้าได้ แต่ต้องถูก audit
- PASS เข้าได้ทันที

6.2 Column-level checks
A. symbol
- type: str
- not null
- in allowed symbol registry
- regex: ^[A-Z0-9_\.\-]+$

B. venue
- type: str
- not null
- allowed list by broker registry

C. source_system
- type: str
- not null
- allowed list: MT5, IMPORTED_MT5

D. timeframe
- type: str
- not null
- fixed to M1 for raw canonical ingest table

E. ts_utc
- type: datetime
- not null
- tz-aware UTC required
- unique within logical key scope
- aligned to 1-minute boundary

F. open
- float64
- not null
- > 0

G. high
- float64
- not null
- > 0

H. low
- float64
- not null
- > 0

I. close
- float64
- not null
- > 0

J. tick_volume
- int64
- not null
- >= 0

K. spread_points
- int32 nullable
- if present, >= 0
- upper anomaly threshold monitored separately

L. real_volume
- int64 nullable
- if present, >= 0

M. ingest_run_id
- str
- not null

N. logical_key
- str
- not null

O. row_hash
- str
- not null
- fixed hash length based on selected hash algorithm

P. batch_fingerprint
- str
- not null

Q. quality_status
- str
- allowed values PASS/WARN/FAIL/QUARANTINE

R. quality_score
- float64
- range [0, 1]

S. continuity_status
- str
- allowed values OK/GAP/DUPLICATE/REPAIRED/CONFLICT

T. conflict_status
- nullable str
- allowed values NONE/HASH_MISMATCH/OHLCV_MISMATCH/LATE_REPLACEMENT

6.3 Cross-column checks
- high >= max(open, close, low)
- low <= min(open, close, high)
- high >= low
- candle_range = high - low >= 0
- body_abs = abs(close - open) <= candle_range
- zero-range candle allowed only as WARN if open==high==low==close and tick_volume==0
- future timestamp beyond ingest time tolerance = FAIL
- ts_utc minute alignment required = FAIL

============================================================
7) THRESHOLD TABLE
============================================================
Thresholds must be dynamic by:
- symbol
- trading session bucket
- weekday
- rolling lookback window
- market regime bucket (later phase)

Base threshold design:
- do not use one global hardcoded threshold for all periods
- use rolling robust statistics

7.1 Rolling windows
Fast window:
- 1 trading day

Short window:
- 5 trading days

Medium window:
- 20 trading days

Long window:
- 60 trading days

7.2 Robust metrics
Use:
- median
- MAD (median absolute deviation)
- percentile bands
- z-like robust score based on MAD

7.3 Monitored signals
Per-row derived metrics:
- candle_range = high - low
- body_abs = abs(close - open)
- upper_wick = high - max(open, close)
- lower_wick = min(open, close) - low
- spread_to_range = spread_points / max(candle_range, epsilon)
- volume_jump_ratio = tick_volume / rolling_median_volume
- range_jump_ratio = candle_range / rolling_median_range
- close_gap_abs = abs(close_t - close_t-1)
- gap_to_range_ratio = close_gap_abs / max(rolling_median_range, epsilon)

7.4 Severity policy
PASS:
- inside expected hard constraints and robust band

WARN:
- passes hard constraints but exceeds soft robust threshold
- row can enter canonical DB with anomaly_flags

FAIL:
- violates deterministic structural rule
- row blocked from canonical DB

QUARANTINE:
- violates severe anomaly or conflict rule
- row stored in quarantine and requires later review/reconciliation

============================================================
8) DEFAULT ANOMALY RULES
============================================================
Rule A: Structural OHLC violation
Condition:
- high < open OR high < close OR high < low
- low > open OR low > close OR low > high
Action:
- FAIL

Rule B: Non-positive prices
Condition:
- any(open, high, low, close) <= 0
Action:
- FAIL

Rule C: ts not aligned to minute
Condition:
- second != 0 or microsecond != 0
Action:
- FAIL

Rule D: future timestamp
Condition:
- ts_utc > ingest_ts_utc + tolerated_clock_skew
Default skew:
- 120 seconds
Action:
- FAIL

Rule E: zero-volume flat bar
Condition:
- open==high==low==close and tick_volume==0
Action:
- WARN
Reason:
- may occur during market inactivity; do not auto-drop

Rule F: extreme range spike
Condition:
- range_jump_ratio > 8 on fast window
Action:
- WARN first
- QUARANTINE if also spread anomaly or conflicting duplicate exists

Rule G: extreme volume spike
Condition:
- volume_jump_ratio > 12 on short window
Action:
- WARN
- QUARANTINE if repeated cluster >= 3 consecutive rows

Rule H: spread explosion
Condition:
- spread_points > rolling_p99_spread * 3
Action:
- WARN
- QUARANTINE if spread_to_range > 0.8

Rule I: duplicate same logical key same row hash
Condition:
- same logical_key and same row_hash already exists
Action:
- mark duplicate
- drop from canonical write
- audit only

Rule J: duplicate same logical key different row hash
Condition:
- same logical_key but row_hash differs
Action:
- QUARANTINE
- conflict_status = HASH_MISMATCH
- do not overwrite canonical automatically

Rule K: continuity gap
Condition:
- expected next minute missing in raw ingest sequence
Action:
- WARN
- continuity_status = GAP
- trigger gap repair workflow later

Rule L: late replacement packet
Condition:
- same ts_utc re-arrives after canonical write with different OHLCV payload
Action:
- QUARANTINE
- conflict_status = LATE_REPLACEMENT

============================================================
9) DEDUP AND UPSERT POLICY
============================================================
9.1 Dedup priority
Priority order:
1. logical_key exact duplicate + same row_hash => ignore write
2. logical_key duplicate + different row_hash => quarantine conflict
3. batch already seen by batch_fingerprint => skip full batch
4. source_file_id + source_row_number duplicate => audit duplicate source

9.2 Canonical write rule
Only rows with:
- PASS
- WARN
and no hash conflict
may enter canonical DB

9.3 Overwrite policy
Canonical row is immutable by default
Exception:
- only approved repair workflow can supersede a quarantined conflict
- every replacement must preserve original row in audit history

9.4 Audit trail
Every decision writes to audit registry:
- run_id
- logical_key
- row_hash
- action_taken
- decision_reason
- previous_row_hash
- new_row_hash
- operator = SYSTEM/AUTO_REPAIR/MANUAL_REVIEW

11) DETERMINISTIC RESAMPLING RULES
============================================================
Derived timeframes:
M2 M3 M4 M5 M6 M10 M15 M30 H1 H4 D1

Resample aggregation:
- open = first(open)
- high = max(high)
- low = min(low)
- close = last(close)
- tick_volume = sum(tick_volume)
- real_volume = sum(real_volume)
- spread_points = max(spread_points) or null-safe aggregate chosen by policy

Resample preconditions:
- source M1 continuity status must be acceptable
- if gap exceeds threshold inside bucket:
  - mark derived bar quality_status = WARN or FAIL by severity
- derived bars keep lineage fields:
  - source_start_ts_utc
  - source_end_ts_utc
  - source_row_count
  - source_gap_count
  - source_quality_rollup

15) QUALITY SCORING
============================================================
Recommended quality score:
Start at 1.0 and subtract penalties

Example penalty model:
- soft anomaly warn: -0.10 each
- continuity gap involvement: -0.20
- spread explosion: -0.15
- range spike: -0.15
- duplicate exact: no canonical write
- hash conflict: quarantine, score 0

Interpretation:
- 0.90 to 1.00 = high trust
- 0.75 to 0.89 = usable with caution
- 0.50 to 0.74 = weak quality
- below 0.50 = quarantine candidate

16) PERFORMANCE PRINCIPLES
============================================================
- Hot path first: normalize -> contract -> dedup -> write
- Avoid heavy reporting in hot path
- Use columnar parquet partitions by symbol/year/month
- Use DuckDB for audits and weekly summaries
- Use incremental rebuild only for impacted time buckets
- Keep contract execution deterministic and reproducible
- Prefer robust statistics over naive mean/std due to fat-tail market data
