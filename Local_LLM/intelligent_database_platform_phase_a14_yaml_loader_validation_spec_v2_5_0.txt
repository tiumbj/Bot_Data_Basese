============================================================
1) FINAL DECISION FOR THIS VERSION
============================================================

LOCK FOR v2.5.0:
1. Registry YAML ทุกไฟล์ต้องถูก validate ด้วย schema model ก่อนใช้งาน
2. Loader ต้องโหลดผ่าน manifest เสมอ
3. Loader ต้องมี checksum verification ทุกไฟล์
4. Loader ต้องเป็น fail-fast ถ้า registry required ใดไม่ผ่าน
5. Runtime ต้องใช้ active immutable registry snapshot
6. Hot reload ต้องเป็น staged -> validated -> activated เท่านั้น
7. ถ้า reload fail ระบบต้อง keep active snapshot เดิม
8. Bot และ service layer ห้ามอ่าน YAML ตรง ต้องอ่านผ่าน registry service/cache เท่านั้น

RATIONALE:
ถ้าปล่อยให้แต่ละ service อ่าน YAML ตรง จะเกิด:
- inconsistent state
- parsing behavior ไม่เหมือนกัน
- schema drift
- path/config chaos
- runtime fail ระหว่าง deploy

ดังนั้นต้องมี registry loader กลางตัวเดียว

============================================================
5) SCHEMA VALIDATION STRATEGY
============================================================

PRIMARY VALIDATOR:
Pydantic models ต่อ YAML แต่ละไฟล์

VALIDATION LEVELS:
L1. File existence validation
L2. YAML parse validation
L3. Pydantic schema validation
L4. Cross-file reference validation
L5. Semantic rule validation
L6. Conflict/overlap validation
L7. Activation readiness validation

DETAIL:
L1 File existence validation
- required files ต้องมีครบ
- path ต้องตรง manifest
- duplicate filename not allowed

L2 YAML parse validation
- parse ได้จริง
- no duplicate keys
- no invalid scalar/type issue

L3 Pydantic schema validation
- field types ถูก
- required field มีครบ
- enum values ถูก
- regex fields ถูก format

L4 Cross-file reference validation
- strategy alias ต้อง map ไป canonical strategy ที่มีอยู่
- timeframe alias ต้อง map ไป canonical timeframe ที่ valid
- file pattern artifact_type ต้องอยู่ใน artifact registry
- metrics aliases ต้อง map ไป canonical metric names เท่านั้น

L5 Semantic rule validation
- priority ต้องไม่ติดลบ
- alias list ห้ามว่างถ้า registry ประเภทนั้น require alias
- canonical ids ต้อง lowercase_snake_case หรือ spec format เดียวกัน
- timeframe minute_value ต้องสอดคล้องกับ label เช่น M15=15, H1=60

L6 Conflict/overlap validation
- pattern overlap ที่ทำให้ ambiguity สูงต้อง fail หรือ quarantine
- alias ซ้ำแบบคนละ canonical target ต้อง fail
- metric alias เดียว map ไปหลาย canonical metrics ไม่ได้
- timeframe alias ซ้ำแบบขัดกันไม่ได้

L7 Activation readiness validation
- required registry loaded ครบ
- checksum pass
- cross references pass
- zero fatal errors
- snapshot assembled ได้ครบ

============================================================
7) CHECKSUM POLICY
============================================================

ALGORITHM:
SHA-256

CHECKSUM UNIT:
byte content ของไฟล์ YAML หลังอ่านแบบ UTF-8 exact bytes

RULE:
manifest ต้องเก็บ checksum ของทุก registry file

VERIFICATION:
1. read file bytes
2. compute sha256
3. compare กับ manifest
4. ถ้า mismatch:
   - fail load
   - mark FAILED_CHECKSUM_MISMATCH
   - move snapshot ไป failed
   - keep active snapshot เดิม

OPTIONAL EXTRA:
สามารถเก็บ normalized-content checksum เพิ่มได้ภายหลัง
แต่ v2.5.0 ให้ exact file bytes เป็นตัวหลักก่อน

WHY:
simple, deterministic, cheap, production-safe

============================================================
8) FAIL-FAST POLICY
============================================================

FATAL CONDITIONS:
1. manifest parse fail
2. missing required registry
3. checksum mismatch
4. schema validation fail ใน required registry
5. cross-file reference fail
6. duplicate canonical id conflict
7. ambiguous pattern overlap ที่ priority แก้ไม่ได้
8. activation snapshot build fail

ON FATAL:
- abort current load
- do not touch current active snapshot
- write failure report
- move staging bundle ไป failed
- emit log + registry_load_event

NON-FATAL CONDITIONS:
1. deprecated alias warning
2. unused optional field warning
3. redundant alias warning
4. unreachable pattern warning
5. stale optional registry warning

ON NON-FATAL:
- activation allowed
- warnings recorded in snapshot metadata

============================================================
9) STAGING -> VALIDATE -> ACTIVATE FLOW
============================================================

STEP-BY-STEP:

STEP 1
อ่าน source YAML จาก config/bridge

STEP 2
copy source bundle เข้า runtime/registry_cache/staging/<bundle_id>

STEP 3
parse manifest และ validate manifest

STEP 4
verify checksums

STEP 5
validate required registries ทีละไฟล์ตาม load order

STEP 6
run cross-file semantic validation

STEP 7
build immutable registry snapshot object

STEP 8
serialize validated snapshot metadata ลง staging

STEP 9
run activation readiness checks

STEP 10
atomic switch:
- previous active -> history
- staging validated snapshot -> active
- update active pointer file

STEP 11
emit REGISTRY_ACTIVATED event

IMPORTANT:
activation ต้องเป็น atomic operation
ห้ามเปลี่ยนทีละไฟล์ใน active directory เพราะจะทำให้ runtime อ่าน snapshot ครึ่งชุด

============================================================
11) HOT-RELOAD POLICY
============================================================

DEFAULT DECISION:
รองรับ hot reload แต่ต้องเป็น controlled reload เท่านั้น

TRIGGERS:
1. manual command
2. operator action
3. scheduled reload
4. CI/CD deploy hook

NOT ALLOWED:
- auto-reload ทุก file change ทันที
- partial reload เฉพาะบาง registry โดยไม่ build ทั้ง bundle

HOT-RELOAD RULE:
1. reload เป็น bundle-level only
2. validate ครบทุกไฟล์ก่อน activate
3. active snapshot เก่ายังคงใช้งานจน activation ใหม่สำเร็จ
4. ถ้า reload fail ให้ rollback อัตโนมัติเป็น no-op ต่อ active snapshot เดิม

WHY:
partial reload เป็นแหล่งของ inconsistent behavior มากที่สุด

============================================================
12) ROLLBACK POLICY
============================================================

ROLLBACK TRIGGERS:
1. activation fail
2. post-activation smoke test fail
3. runtime incompatibility detected
4. operator emergency rollback

ROLLBACK METHOD:
- pointer-based rollback
- เปลี่ยน active pointer กลับไปยัง snapshot ล่าสุดที่ stable
- ห้าม regenerate snapshot ตอน rollback

ROLLBACK REQUIREMENTS:
1. history ต้องเก็บอย่างน้อย 20 activated bundles
2. ทุก history entry ต้องมี activation_report
3. rollback action ต้อง emit REGISTRY_ROLLBACK event
4. rollback reason ต้องถูกบันทึกเสมอ

14) OVERLAP AND AMBIGUITY POLICY
============================================================

FILE PATTERN OVERLAP:
ต้องตรวจว่า pattern สองตัวจับชื่อไฟล์กลุ่มเดียวกันแล้วให้คนละ artifact meaning หรือไม่

RESOLUTION RULE:
1. exact pattern specificity ชนะ broad pattern
2. higher priority ชนะ lower priority
3. ถ้ายังเสมอ = fatal ambiguity

ALIAS AMBIGUITY:
ถ้า alias เดียวไปได้หลาย canonical target:
- required registry = fatal
- optional extension registry = quarantine

METRIC COLLISION:
เช่น:
- "winrate"
- "win_rate"
- "wr"
ทั้งหมด map ไป canonical เดียวได้
แต่ห้าม alias เดียว map ไปทั้ง "win_rate_pct" และ "win_count"

============================================================
16) REGISTRY HEALTH MODEL
============================================================

HEALTH STATES:
1. HEALTHY
2. HEALTHY_WITH_WARNINGS
3. DEGRADED
4. FAILED

CONDITION EXAMPLES:
HEALTHY
- active snapshot loaded
- no fatal
- zero or low warnings

HEALTHY_WITH_WARNINGS
- active snapshot loaded
- warnings present
- runtime usable

DEGRADED
- active snapshot old/stale
- last reload failed
- fallback active still usable

FAILED
- no active snapshot at startup
- registry service boot failed
- cannot resolve required registries

SERVICE RULE:
bot-facing API ควรยังตอบได้ถ้า state = DEGRADED
แต่ถ้า FAILED ต้อง block decision paths ที่ต้องพึ่ง registry

============================================================
17) STARTUP POLICY
============================================================

STARTUP ORDER:
1. load settings
2. initialize logger
3. start registry loader
4. verify active snapshot exists
5. if no active snapshot:
   - try full load from source bundle
6. if load success:
   - boot services
7. if load fail and no previous active snapshot:
   - startup fail hard
8. if load fail but previous active snapshot exists:
   - boot in DEGRADED mode with active snapshot เดิม

WHY:
production ต้องไม่สะดุดเพราะ source YAML ใหม่เสีย
ถ้ายังมี active snapshot เดิม ระบบควรเดินต่อได้

19) VERSIONING POLICY
============================================================

SEMVER RULE:
bundle_semver ใช้ semantic versioning

MAJOR:
breaking registry contract

MINOR:
new additive aliases/patterns/optional fields

PATCH:
fix typo/checksum/documentation/priority adjustment without breaking interface

RULE:
canonical ids ที่ publish แล้ว ห้ามเปลี่ยนชื่อแบบ breaking ใน patch/minor

- ทำ migration note
- เลื่อนเป็น major

5. registry bundle ที่ไม่มี history backup

ALLOWED HOT-RELOAD WINDOW:

- scheduled low-risk window
