1) LOCKED FOUNDATIONS FROM UPLOADED REFERENCE
====================================================================
1.1 Core architecture lock
- Python ไม่ใช่ market analyst หลัก
- Python ทำหน้าที่ data transport, feature build, storage, retrieval, schema validation, risk guard, MT5 execution
- Local LLM ทำหน้าที่ market analyst, final confirm, exit guardian
- Historical backtest + labels = central memory layer
- Similar-case retrieval ต้องคัดเคสก่อนส่งเข้า LLM
- Execution และ hard risk อยู่ฝั่ง Python/MT5 เท่านั้น

1.2 Objective lock
เป้าหมายหลักไม่ใช่แค่เพิ่ม win rate
เป้าหมายหลักคือ:
- ลด full SL frequency
- ลด average loss per losing trade
- เพิ่ม early protective exit
- เพิ่ม scratch / small-loss exit
- รักษา expectancy ให้บวก
- ลดการเข้า trade ใน regime ที่เสียเปรียบ

1.3 Research lock
ทุก strategy ต้อง backtest ทุก research timeframe แบบบังคับ
Timeframes ที่ต้องทดสอบ:
M1, M2, M3, M4, M5, M6, M10, M15, M30, H1, H4, D1

1.4 Micro-exit lock
- Micro exit เป็น base layer บังคับ
- ต้อง backtest ทุก TF
- ต้องวัดผลแบบ cross-timeframe
- ห้ามข้ามการวิจัยส่วนนี้

1.5 Toolchain lock
- Python
- Polars
- Parquet
- Custom Numba-based backtest engine
- Ray สำหรับ parallel research scale-out
- DuckDB สำหรับ analytical central DB
- Qdrant สำหรับ similar-case retrieval
- Optional: VectorBT-style research idea ใช้เป็น reference ได้ แต่ core engine ต้องควบคุมได้เอง

1.6 Label/memory lock
ต้องเก็บทั้งไม้ชนะและไม้แพ้
Label ขั้นต่ำที่ต้องมี:
- win
- loss
- scratch
- fail_fast
- early_exit_good
- early_exit_bad
- trend_follow_success
- fake_breakout_fail
- momentum_collapse
- regime_conflict_fail
- should_exit_early
- fail_after_fake_breakout
- fail_after_regime_shift
- fail_after_momentum_collapse
- fail_due_to_htf_conflict
- recover_after_deep_pullback

1.7 Evaluation lock
Metrics ขั้นต่ำ:
- net profit
- profit factor
- expectancy
- max drawdown
- average trade
- median trade
- win rate
- payoff ratio
- ulcer index
- time in trade
- exposure
- trade density
- stability across windows
- out-of-sample robustness
- transition damage
- parameter plateau width

Winner rule:
- positive expectancy
- acceptable drawdown
- sufficient trade count
- stable across windows
- robust out-of-sample behavior
- not destroyed during transitions
- plateau stability preferred over point optimum

2) SYSTEM DESIGN: STANDALONE BACKTEST + CENTRAL DATABASE
====================================================================
2.1 Design goal
ออกแบบให้ระบบ backtest และ database เป็น standalone research core
ที่ downstream bot ทุกตัวสามารถเข้าถึงข้อมูลกลางได้แบบ real-time หรือ near real-time

2.2 Main principles
- Single source of truth for historical market data
- Single source of truth for labels / memory / winners registry
- Standardized schema across backtest, live monitor, and bot execution
- Versioned everything: data, feature, label, schema, prompt, strategy, backtest
- Bots ห้ามสร้าง logic ความจริงของตัวเองคนละแบบ
- Bots ต้องอ่านจาก central outputs ที่ผ่าน research แล้ว

2.3 Standalone architecture
Layer A: Data Ingestion Layer
- ดึง historical + incremental bars จาก MT5
- Normalize symbol / timeframe / timestamp
- Deduplicate data
- Store raw and canonical data

Layer B: Feature / Regime Layer
- สร้าง features ข้าม TF
- สร้าง regime labels
- สร้าง structure / momentum / volatility / session states

Layer C: Backtest / Replay Layer
- run entry/exit simulations
- run TP/SL surface research
- run walk-forward
- run transition damage analysis
- run micro-exit research

Layer D: Label / Memory Layer
- สร้าง outcome labels
- สร้าง failure pattern labels
- สร้าง case memory units
- สร้าง retrieval-ready snapshots

Layer E: Registry / Publish Layer
- publish winners
- publish current approved packages
- publish latest labels / memory / retrieval index
- publish ready snapshot ให้ downstream bots ใช้

Layer F: Bot Consumption Layer
- bots query central DB / registry / retrieval layer
- bots ดึง approved logic + latest statistics + similar case context
- execution bots ใช้ Python decision engine + local LLM confirm/exit guardian

2.4 Why standalone
ข้อดี:
- แยก research ออกจาก execution
- ไม่ทำให้ bot จริงรับภาระ backtest หนัก
- ทำ nightly/weekly jobs ได้คงที่
- ทำ versioning และ rollback ง่าย
- bot หลายตัวใช้ข้อมูลกลางเดียวกันได้
- ง่ายต่อการพิสูจน์ว่าผลลัพธ์มาจาก strategy version ไหน

4) STRATEGIES / LOGIC
====================================================================
4.1 Base logic family
Base entry family ที่ล็อกไว้:
- Market Structure / BOS / CHOCH
- Pullback
- ATR
- ADX
- EMA stack / EMA filters
- Swing logic
- Deep pullback variants
- CHOCH confirm variants
- EMA tight / BOS strict variants

4.2 Core philosophy
- ใช้ Price Action + Structure เป็นแกนหลัก
- ใช้ ATR / ADX / EMA เป็น filter ไม่ใช่แกนทำนายทั้งหมด
- ใช้ regime-aware package selection
- 1 regime ไม่จำเป็นต้องใช้ strategy package เดียวกับอีก regime

4.3 Regime playbook principle
ตัวอย่าง:
- Regime A = entry set A1 + micro exit M1
- Regime B = entry set B2 + micro exit M3
- Regime D = trend-follow set D1 + trailing logic
- Regime F = breakout expansion set F1 + volatility stop

4.4 Entry logic scope
ต้องวิจัย logic ต่อไปนี้ให้ครบ:
- trend continuation
- pullback continuation
- deep pullback
- swing confirmation
- breakout expansion
- breakout failure / fake breakout
- reversal after exhaustion
- structure reclaim / failed reclaim
- HTF alignment / HTF conflict
- session-sensitive entries

4.5 Exit logic scope
ต้องวิจัยให้ครบ:
- base exit
- fast invalidation
- momentum fade
- short EMA weakness / trend weakness
- time stop
- trailing variants
- volatility stop
- structure break exit
- partial exit
- tighten stop
- micro exit early kill

4.6 Exit intelligence principle
Exit ไม่ควรรอแค่ TP/SL
ต้องประเมิน:
- edge ยังอยู่ไหม
- trade health เสื่อมหรือยัง
- multi-TF alignment ยังอยู่ไหม
- pattern เริ่มเหมือน historical failed cases หรือไม่
- ควร hold / tighten / partial / exit_now

10) FIRST BOT DESIGN USING THIS DATABASE
====================================================================
10.1 Bot sequence
1. First bot ดึง approved strategy / registry / labels / retrieval context จาก central system
2. Python engine หา candidate signal จาก approved backtest logic
3. Local LLM รับ realtime multi-TF summary + similar backtest cases
4. Local LLM ส่ง final confirm กลับมาแบบ strict JSON
5. Python engine ตรวจ schema + confidence + hard risk แล้วคำนวณ Entry / TP / SL / RR / lot
6. Python ส่ง order ไป MT5
7. Python เก็บ order context ลง DB
8. Python monitor order ด้วย order/ticket number แบบตายตัว
9. Micro exit monitor ทำงานถี่ระดับประมาณ 1.0 วินาทีได้ ถ้า latency และทรัพยากรรับได้
10. Python เก็บ exit reason และ exit context ลง DB
11. เมื่อ market ปิด ระบบ backtest / labeling / memory update ทำงานเพื่อเตรียมรอบถัดไป

10.2 Order data to store

18) FINAL SINGLE-BEST-PRACTICE CONCLUSION
====================================================================
แนวทางที่ถูกต้องที่สุดสำหรับโปรเจกต์นี้คือ:

สร้าง central standalone research core ที่ใช้
Polars + Parquet + DuckDB + Custom Numba Backtest + Ray + Qdrant

จากนั้นบังคับให้ทุก strategy ถูกทดสอบทุก TF
โดยมี micro exit เป็น base layer
เก็บผลลัพธ์ทั้งหมดเป็น central labeled memory

แล้วให้ bot ตัวแรกใช้ flow นี้:
Python engine หา candidate setup
-> ดึง realtime multi-TF summary
-> ดึง similar backtest cases
-> ส่งให้ local LLM ทำ final confirm / exit guardian
-> Python ตรวจ hard risk + คำนวณ execution
-> MT5 ส่ง order
-> ผลลัพธ์ถูกเก็บกลับเข้าสู่ central DB เพื่อเรียนรู้รอบถัดไป

เอกสารนี้ต้องถูกใช้เป็น master text กลางของโปรเจกต์
และต้องอัปเดตทุกครั้งหลังจบแต่ละขั้นตอน
พร้อมผลลัพธ์และ strategy/logic used
