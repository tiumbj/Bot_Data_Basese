SECTION 1 — FINAL DECISION FOR LAYER 2
============================================================
FINAL DECISION:
- ใช้ Great Expectations (GX Core) เป็น Layer 2 Data Contract Gate
- ใช้ Polars + DuckDB เป็น deterministic row-level dedup / canonicalization engine
- เก็บ Soda เป็น OPTIONAL future layer สำหรับ observability / anomaly monitoring ภายหลัง
  ไม่ใช้เป็นแกนหลักของ Layer 2 ใน phase ปัจจุบัน

WHY THIS IS THE BEST CURRENT DECISION:
A. Layer 2 ของระบบนี้ต้องทำงาน embedded ใน Python pipeline แบบ local-first
   และต้องคุม quarantine / unexpected rows / contract violation ได้ละเอียด
B. GX Core เหมาะกับการฝังเข้า Python orchestration และรองรับ expectation-based validation,
   custom SQL expectation, และ unexpected rows workflow
C. Soda ปัจจุบันแข็งแรงมากในแนว data contracts + observability + anomaly monitoring
   แต่ในงานนี้ Layer 2 ต้องการ deterministic ingestion gate ก่อนเข้า Canonical DB
   ซึ่งยังเหมาะกับ GX + custom Polars/DuckDB มากกว่า
D. Dedup ที่เร็วและ deterministic ไม่ควรพึ่ง data-quality framework โดยตรง
   ต้องให้ Polars/DuckDB ทำ แล้วให้ GX ทำหน้าที่ verify + quarantine + contract reporting

LOCKED PRINCIPLE:
- Data quality framework = contract gate
- Dedup engine = Polars/DuckDB
- Canonical DB receives only PASS rows
- FAIL / WARN rows go to quarantine with diagnostics

SECTION 2 — TECHNOLOGY REVIEW SUMMARY
============================================================
2.1 GREAT EXPECTATIONS (GX)
STRENGTHS:
- Python-native integration
- expectation-based contracts
- custom SQL expectation support
- unexpected rows extraction / failed-row investigation
- validation history model
- suitable for embedded pipeline execution
- suitable for contract-first ingestion gate

BEST USE IN THIS PROJECT:
- schema contract
- null / range / uniqueness / timestamp / OHLC logic contract
- batch validation before canonical write
- quarantine routing
- validation artifact creation
- row sample / unexpected rows diagnostics

2.2 SODA
STRENGTHS:
- strong data contracts direction
- strong observability direction
- anomaly detection / monitors
- convenient declarative checks
- useful for team-facing quality monitoring

BEST USE IN THIS PROJECT:
- future observability layer
- weekly intelligence monitoring
- freshness / drift / anomaly dashboard
- multi-dataset health monitoring
- service governance layer after platform stabilizes

2.3 WHY NOT USE SODA AS PRIMARY LAYER 2 RIGHT NOW
- Layer 2 here is not just “quality checks”; it is an ingestion gate tightly coupled to
  row-level canonicalization and quarantine logic
- We need Python-embedded control, deterministic pass/fail routing, and close integration
  with Polars/DuckDB transforms
- We want one minimal critical-path dependency at ingestion time
- Therefore GX fits better as the contract gate, while Soda remains a strong optional add-on later

SECTION 4 — WHAT GREAT EXPECTATIONS MUST CHECK AT LAYER 2
============================================================
4.1 REQUIRED CONTRACTS FOR OHLCV
A. STRUCTURE CONTRACT
- required columns exist
- correct column order not mandatory, but required names mandatory
- timestamp column exists
- open/high/low/close numeric
- tick_volume numeric or integer-like
- spread optional but if present must be numeric
- real_volume optional but if present must be numeric

B. TYPE CONTRACT
- timestamp parseable to UTC-aligned datetime
- price columns castable to float64
- volume castable to int64/float64 as defined
- no string contamination in numeric fields

C. KEY CONTRACT
- no null timestamp
- no null open/high/low/close
- symbol must match expected symbol alias registry
- timeframe source must match ingestion config

D. OHLC LOGIC CONTRACT
- high >= max(open, close, low)
- low <= min(open, close, high)
- high >= low
- open > 0
- high > 0
- low > 0
- close > 0

E. RANGE / STATISTICAL CONTRACT
- bar_range = high - low must be >= 0
- zero-range bars allowed only within policy threshold
- extreme outlier bars flagged if z-score / robust MAD threshold breached
- impossible jumps beyond configured sanity boundary flagged
- negative volume forbidden
- suspicious repeated identical OHLC bars flagged

F. TEMPORAL CONTRACT
- timestamp monotonic in batch after sort
- timestamp must align to expected minute boundary
- future timestamps forbidden
- duplicates by timestamp flagged
- gap count recorded
- overlap count recorded

G. SOURCE CONTRACT
- source broker id present
- source batch id present
- ingestion ts present
- file fingerprint present
- pipeline version present

4.2 CONTRACT OUTCOMES
- PASS: batch can enter staging_pass
- WARN: batch enters staging_warn and continues only if policy allows
- FAIL: rows or batch routed to quarantine_rows and canonical write blocked for affected rows
- HARD FAIL: job stops and marks run as failed

SECTION 5 — IMPORTANT DESIGN RULE
============================================================
Great Expectations must NOT be the tool that performs final dedup.
It verifies conditions and surfaces unexpected rows.
Final duplicate resolution must be handled by deterministic code in Polars/DuckDB.

REASON:
- faster
- easier to benchmark
- easier to reason about precedence
- easier to reproduce
- easier to keep exact same results across weekly runs

============================================================
SECTION 6 — DEDUP ENGINE SPEC
============================================================
6.1 PRIMARY KEYS
PRIMARY LOGICAL KEY:
(symbol_canonical, timeframe_source, ts_utc)

TECHNICAL KEYS:
- row_hash_raw
- row_hash_normalized
- batch_fingerprint
- source_record_id if available

6.2 DEDUP ORDER
STEP 1: Remove exact duplicates by row_hash_normalized
STEP 2: Detect logical duplicates by (symbol_canonical, timeframe_source, ts_utc)
STEP 3: If multiple rows remain for same logical key, resolve by precedence:
        a. higher source_priority
        b. later ingestion_ts
        c. more complete metadata
        d. lower anomaly_score
        e. else route to quarantine_conflict
STEP 4: Write canonical survivor
STEP 5: Write duplicate ledger for all dropped rows

6.3 DUPLICATE LEDGER MUST STORE
- run_id
- symbol
- timeframe_source
- ts_utc
- survivor_row_hash
- dropped_row_hash
- duplicate_type
- resolution_reason
- source_priority
- source_batch_id
- ingestion_ts

SECTION 7 — STATISTICAL ANOMALY SCREEN AT LAYER 2
============================================================
Use LIGHTWEIGHT statistical anomaly checks before Canonical DB.
This is not full observability yet; this is pre-canonical safety screening.

REQUIRED CHECKS:
- robust z-score of bar range
- robust z-score of body size
- repeated identical OHLC streak count
- zero-volume abnormality
- impossible timestamp density
- abnormal gap cluster
- broker-switch discontinuity
- weekend/holiday abnormal insertions

POLICY:
- anomaly score below warn threshold -> PASS
- anomaly score between warn and fail threshold -> WARN
- anomaly score above fail threshold -> QUARANTINE

IMPORTANT:
These anomaly checks should be computed in Polars first for speed.
GX then validates the resulting derived fields / thresholds as part of the contract decision.

SECTION 18 — FINAL LOCK STATEMENT
============================================================
LOCK STATEMENT:
For this project, Layer 2 shall use Great Expectations as the embedded Data Contract Gate,
while Polars + DuckDB remain the deterministic engine for normalization, anomaly scoring,
row-level dedup, conflict resolution, and canonical writing.
Soda is reserved as an optional future observability layer, not the primary critical-path
contract gate for canonical ingestion in the current platform phase.
