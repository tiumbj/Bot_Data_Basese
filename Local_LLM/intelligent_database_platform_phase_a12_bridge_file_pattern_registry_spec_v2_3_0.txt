================================================================
0) EXECUTIVE DECISION
================================================================

เอกสารนี้ล็อกว่า output ที่มีอยู่แล้วจากระบบ test/research ปัจจุบัน
ต้องถูกดูดเข้า platform กลางด้วย "Bridge Registry Layer"
โดยใช้หลักการ 5 ข้อ:

1. additive-only
   - ห้ามรื้อ pipeline เดิม
   - ห้ามทำให้ build/test ที่กำลังรันอยู่หยุด

2. deterministic parsing
   - ชื่อไฟล์ / path / folder / metric alias ต้องถูก parse แบบกติกาเดียว
   - ห้ามตีความจากข้อความลอย ๆ แบบไม่คงที่

3. identity-first
   - ทุก artifact ต้องสร้าง identity กลางได้
   - ใช้ run_id / artifact_id / strategy_fingerprint / timeframe / dataset_id

4. normalization-before-publish
   - ไฟล์เดิมใช้ต่อได้
   - แต่ก่อนเข้า registry กลาง ต้อง normalize schema ก่อนเสมอ

5. traceable lineage
   - ต้องไล่ย้อนหลังได้ว่า artifact นี้มาจากไฟล์ไหน
   - มาจาก command ไหน
   - ใช้ data window ไหน
   - ใช้ logic/strategy อะไร

================================================================
1) GOAL OF A12
================================================================

A12 มีเป้าหมาย 6 ข้อ:

1. กำหนด file pattern registry กลาง
2. กำหนด parser rules สำหรับชื่อไฟล์และ path
3. กำหนด field mapping เข้าสู่ registry schema
4. กำหนด metric alias normalization
5. กำหนด promotion rules จาก legacy output -> normalized artifact
6. ทำให้ผล test เดิมกลายเป็นสินทรัพย์กลางที่ bot ใช้ต่อได้

================================================================
2) SCOPE OF FILES TO INGEST
================================================================

Bridge layer ต้องรองรับ artifact อย่างน้อย 10 กลุ่ม:

A. package build outputs
B. entry component package outputs
C. micro exit outputs
D. timeframe benchmark outputs
E. walk-forward validation outputs
F. parameter sweep outputs
G. summary txt/csv/parquet/json outputs
H. run logs
I. strategy note / decision note / checklist note
J. publish candidate manifests

ไฟล์อาจอยู่ในรูปแบบ:
- .txt
- .json
- .jsonl
- .csv
- .parquet
- .log

================================================================
3) LEGACY OUTPUT CLASSIFICATION
================================================================

Bridge layer ต้องจัดกลุ่มไฟล์เดิมเป็น 4 ระดับ:

LEVEL 1: RAW_RESEARCH_ARTIFACT
- output ดิบจากการ run
- ยังไม่ normalize
- ยังไม่มี metric schema มาตรฐาน

LEVEL 2: PARSED_RESEARCH_ARTIFACT
- parse file name / path / metadata ได้แล้ว
- แต่อาจยังไม่ normalize metrics ทั้งหมด

LEVEL 3: NORMALIZED_RESEARCH_RECORD
- map เข้า field กลางได้ครบ
- มี strategy_id, timeframe, dataset_id, metrics standard แล้ว

LEVEL 4: PACKAGE_CANDIDATE_RECORD
- พร้อมใช้ใน package registry
- มี coverage evidence
- มี validation evidence
- มี micro-exit evidence ถ้าต้องใช้

5) FILE PATTERN REGISTRY STANDARD
================================================================

ให้มี registry กลาง 1 ไฟล์ เช่น:

config\bridge_file_patterns.yaml

โครงมาตรฐานต่อ 1 pattern:

- pattern_id
- artifact_family
- glob_pattern
- regex_pattern
- priority
- enabled
- parser_name
- metric_profile
- promotion_target
- notes

ตัวอย่าง concept:

pattern_id: entry_component_package_json
artifact_family: entry_component_package
glob_pattern: "**/*entry*component*package*.json"
regex_pattern: "(?P<strategy>.+?)_(?P<timeframe>M1|M2|M3|M4|M5|M6|M10|M15|M30|H1|H4|D1)_(?P<kind>package|summary|validation)"
priority: 100
enabled: true
parser_name: parse_entry_component_package_json
metric_profile: standard_backtest_metrics_v1
promotion_target: normalized_research_record

================================================================
6) FILE DISCOVERY RULES
================================================================

Bridge scanner ต้องทำตามลำดับนี้:

STEP 1
scan root paths ตาม config

STEP 2
คัดเฉพาะ file extensions ที่รองรับ

STEP 3
match pattern ด้วย glob ก่อน
- เพื่อความเร็ว

STEP 4
validate ด้วย regex_pattern
- เพื่อดึง named groups

STEP 5
เรียก parser ตาม parser_name

STEP 6
สร้าง parsed artifact metadata

STEP 7
normalize metrics/schema

STEP 8
upsert เข้า registry

กฎ:
- ถ้าไฟล์ match หลาย pattern ให้เลือก priority สูงสุด
- ถ้า priority เท่ากัน ให้ quarantine เป็น ambiguous_pattern_match
- ห้าม parse ไฟล์เดิมซ้ำถ้า content hash เดิมและ mtime เดิม

8) CANONICAL ARTIFACT IDENTITY
================================================================

ทุกไฟล์ที่ ingest ต้องสร้าง identity กลางดังนี้:

artifact_id =
hash(
  normalized_abs_path +
  file_size +
  file_mtime_utc +
  content_hash_sha256_prefix
)

record_id =
hash(
  artifact_family +
  strategy_fingerprint +
  timeframe +
  dataset_id +
  test_window_start +
  test_window_end +
  metric_profile +
  legacy_source_tag
)

run_id:
- ถ้า payload มี run_id ให้ใช้
- ถ้าไม่มี ให้ bridge สร้าง synthetic_run_id
  จาก command/file group/time window

strategy_fingerprint =
hash(
  normalized_strategy_name +
  entry_logic_signature +
  exit_logic_signature +
  filter_signature +
  package_variant_signature
)

================================================================
9) STRATEGY NAME NORMALIZATION
================================================================

ปัญหาหลักของ output เดิมคือ strategy naming มักยาวและไม่คงที่
จึงต้อง normalize เป็น 3 ชั้น:

A. raw_strategy_name
- ชื่อเดิมทั้งหมดจากไฟล์

B. normalized_strategy_name
- lowercase
- แทนช่องว่างด้วย _
- ตัดอักขระพิเศษที่ไม่จำเป็น
- collapse "__" เป็น "_"

C. strategy_family_key
- map alias หลายชื่อที่จริงคือกลุ่มเดียวกัน
- เช่น:
  pullback_deep
  entry_v2_pullback_deep
  locked_pullback_deep
  -> strategy_family_key = pullback_deep

- quarantine reason = timeframe_unresolved

12) TEST WINDOW EXTRACTION RULE
================================================================

Bridge layer ต้องพยายามดึง test window จาก 3 แหล่งตามลำดับ:

PRIORITY 1

PRIORITY 2

PRIORITY 3
ชื่อไฟล์/path

- test_window_start = null
- test_window_end = null

- quarantine เป็น ambiguous_test_window

================================================================
16) PROMOTION TARGETS
================================================================

ผลจาก parser จะถูก promote ตามเงื่อนไข:

16.1 RAW -> PARSED
เมื่อ pattern match + parser คืนค่า parse_status=ok

16.2 PARSED -> NORMALIZED
เมื่อ:
- timeframe resolved
- strategy resolved
- metric profile resolved
- dataset_id resolved
- schema minimum pass

16.3 NORMALIZED -> PACKAGE_CANDIDATE
เมื่อ:
- metrics core ครบขั้นต่ำ
- test window ชัด
- strategy_fingerprint ชัด
- timeframe coverage evidence ผ่าน
- validation evidence ผ่านตาม policy

================================================================

Bridge layer ต้องสะสม evidence ว่า strategy family ใด coverage ครบแล้ว

coverage matrix:
- strategy_family_key
- timeframe
- entry_test_present
- exit_test_present
- micro_exit_present
- validation_present
- latest_record_id

ถ้า TF ใดขาด:
- strategy ยัง publish เป็น "fully covered" ไม่ได้

18) MICRO EXIT EVIDENCE RULE
================================================================

ทุก strategy/package ที่จะใช้ production research ต้องมีสถานะหนึ่งในนี้:

- micro_exit_verified

สำหรับโปรเจกต์นี้ให้ถือว่า:
- micro_exit_missing = ไม่ผ่าน publish final

================================================================
20) DUPLICATE HANDLING RULES
================================================================

แยก duplicate เป็น 3 แบบ:

TYPE A: duplicate_artifact
- ไฟล์เดิมแท้ ๆ
- path/content hash เดิม
- skip ได้

TYPE B: duplicate_record_same_metrics
- คนละไฟล์ แต่ normalized record เดียวกัน
- mark alias_of เดิม

TYPE C: duplicate_record_conflict
- identity เดียวกัน แต่ metric ไม่เท่ากัน
- quarantine และต้อง review
