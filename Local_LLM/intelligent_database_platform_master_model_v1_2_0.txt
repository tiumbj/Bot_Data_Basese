============================================================
1) LOCKED FOUNDATIONS
============================================================

1.1 Architecture lock
- ต้องแยกฐานข้อมูล/backtest ออกจาก bot runtime
- Database Platform เป็นสินทรัพย์กลาง
- Bot Runtime เป็น client ของ platform กลาง

1.2 Data stack lock
- Python = orchestration layer
- Polars = data processing layer
- Parquet = primary storage format
- DuckDB = analytical query engine / metadata registry / SQL analytics
- Qdrant = similar-case retrieval layer
- Custom Numba-based backtest engine = research core
- Ray = parallel scaling layer

1.3 Research lock
- ต้องทดสอบทุก single timeframe แบบบังคับ:
  M1, M2, M3, M4, M5, M6, M10, M15, M30, H1, H4, D1
- micro exit เป็น base layer บังคับทุก timeframe
- ต้องมี in-sample / out-of-sample / walk-forward / regime-stratified validation
- ห้ามเลือกผู้ชนะจาก point optimum แคบ
- ต้องเลือก plateau stability ที่ robust

1.4 Label lock
- Triple-Barrier
- Meta-Labeling
- custom exit/failure labels
- transition/regime damage labels
- execution quality labels

1.5 Bot integration lock
- Local LLM ต้องไม่รับ historical raw data ทั้งหมดตรง ๆ
- ต้อง query similar historical cases จาก Qdrant ก่อน
- Python engine ต้องเป็น final hard-risk / execution correction layer
- MT5 execution แยกจาก reasoning layer

2) SYSTEM SPLIT
============================================================

2.1 Intelligent Database Platform
หน้าที่:
- ingest MT5 market data
- build canonical datasets
- build feature store
- run backtests
- generate labels
- generate case memory
- maintain retrieval index
- maintain strategy registry
- maintain weekly intelligence snapshots
- publish data service/API to bots

- compare live outcomes vs research expectation
- learn from results incrementally

2.2 Bot Runtime Platform
หน้าที่:
- receive realtime market feed
- build live multi-TF snapshot
- ask platform for:
  - current regime summary
  - approved strategy packages
  - approved micro exit package
  - similar historical cases
  - risk warnings
- send context to local LLM
- receive final confirm
- correct execution terms
- send order to MT5
- monitor order by ticket id
- trigger micro exit / protective logic
- send final trade outcome back to platform

4) STORAGE ARCHITECTURE
============================================================

4.1 Primary storage
Parquet เป็น storage หลัก เพราะ:
- columnar
- compressed
- scan/filter/read เร็ว
- เหมาะกับ analytical workload
- ใช้ร่วมกับ Polars และ DuckDB ดีมาก

4.2 Analytical database
DuckDB ใช้สำหรับ:
- query ข้าม TF
- query ข้าม strategy package
- query ข้าม regime
- compare runs
- maintain registries
- create SQL views ให้ bot เรียกง่าย

4.3 Vector retrieval database
Qdrant ใช้เฉพาะ similarity layer:
- similar historical cases
- filter ตาม symbol / TF stack / regime / session / direction
- ส่ง top-k cases ให้ local LLM
- ห้ามใช้เป็น main analytical DB

7) WEEKLY INTELLIGENCE RUN DESIGN
============================================================

เมื่อกด run ประจำสัปดาห์ ระบบต้องทำตามลำดับนี้:

Step 1: Ingest new MT5 data
- ดึง bars ใหม่จาก MT5
- check source hash
- check timestamp continuity
- reject duplicates
- log ingest batch

Step 2: Normalize to canonical dataset
- map symbol
- align timezone
- validate OHLC integrity
- write canonical parquet partitions

Step 3: Rebuild affected timeframes
- build only impacted partitions by default
- full rebuild only when source corruption or rule change exists

Step 4: Update feature store
- recalc affected features
- maintain version tag

Step 5: Run backtest batch
- run approved research jobs
- run pending package jobs
- run micro exit base jobs across all TF

Step 6: Run validation batch
- IS / OOS / walk-forward
- regime-stratified validation
- transition damage analysis

Step 7: Update labels
- triple-barrier
- meta-label
- custom failure / exit labels
- execution quality labels

Step 8: Update case memory
- create new cases
- refresh summaries if label version changed

Step 9: Update retrieval index
- generate embeddings
- upsert to Qdrant
- remove stale/replaced cases if needed

Step 10: Update package registry
- promote winners
- demote damaged packages
- keep full history

Step 11: Compare live trades vs research expectation
- compare slippage
- compare regime mismatch
- compare live loss clusters
- identify drift

Step 12: Publish intelligence outputs
- weekly report
- active package registry
- regime mapping summary
- micro exit recommendation set
- bot service views

Step 13: Update project texts
- progression checklist
- step result note
- strategy/logic use note

============================================================
12) BOT DESIGN USING THE PLATFORM
============================================================

Bot flow:

1. Bot gets realtime market data
2. Python engine builds live snapshot
3. Python asks platform for approved package + similar cases + regime summary
4. Local LLM receives:
   - live snapshot
   - approved package candidates
   - top similar labeled cases
   - current warnings
5. Local LLM returns analysis / confirm / reject
6. Python corrects entry, TP, SL, RR, lot, execution safety
7. MT5 sends order
8. Python stores live order data to central platform
9. Python monitors order by order ticket
10. micro exit checks every fast loop interval
11. Python stores exit reason and outcome
12. next weekly run consumes these outcomes into intelligence layer
