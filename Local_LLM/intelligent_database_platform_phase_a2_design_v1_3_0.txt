============================================================
1) PHASE A2 SCOPE
============================================================

เป้าหมายของ Phase A2 มี 6 เรื่องเท่านั้น

A2.1 Canonical Ingest Pipeline
- รับข้อมูล price/source เข้าระบบกลาง
- normalize ให้เป็นรูปแบบเดียว
- ทำให้ทุก downstream job ใช้ schema เดียวกัน

A2.2 Duplicate Prevention
- กันข้อมูลซ้ำตั้งแต่ source ingest
- กันซ้ำอีกชั้นตอน append/update
- เก็บ fingerprint/hash เพื่อ audit ได้

A2.3 Multi-Timeframe Canonical Builder
- ใช้ M1 เป็น source หลัก
- สร้างทุก TF ที่ล็อกไว้แบบ deterministic
- rebuild เฉพาะช่วงที่ได้รับผลกระทบได้

A2.4 Weekly Intelligence Run
- เมื่อกด run ระบบต้องทำรอบอัปเดตกลางทั้งชุด
- เก็บผลลัพธ์, labels, memory, package registry, snapshot

A2.5 Central Registry + Metadata
- มี registry กลางคุม dataset, features, labels, packages, runs
- version ทุกอย่าง
- trace ได้ว่า bot ใช้ข้อมูลรุ่นไหน

A2.6 Service Contract for Other Bots
- bot ตัวอื่นเรียกใช้ข้อมูลกลางได้
- ไม่ต้องสร้าง backtest/label/retrieval ซ้ำเอง

============================================================
2) LOCKED PRINCIPLES
============================================================

2.1 Single source of truth
- Canonical M1 dataset เป็น source กลาง
- ทุก TF downstream ต้อง derive จาก canonical M1 เท่านั้น

2.2 Deterministic rebuild
- ถ้าข้อมูลต้นทางเหมือนเดิม ต้องสร้าง output เดิมได้
- ห้ามมี logic แบบสุ่มโดยไม่ล็อก seed

2.3 Idempotent pipeline
- กด run ซ้ำได้ โดยไม่ทำให้ข้อมูลเสีย
- run ซ้ำแล้วไม่ควรเกิด duplicate rows/result pollution

2.4 Central intelligence
- labels, memory cases, strategy winners, regime summaries
  ต้องเป็นทรัพย์สินกลางของ platform
- bot ห้ามสร้างเวอร์ชันของตัวเองซ้ำ

2.5 Auditability
- ทุกการ ingest/build/backtest/label/update ต้อง trace ได้
- ต้องตอบได้ว่า output นี้มาจาก input ชุดไหน + logic version ไหน

2.6 Weekly evolution
- ระบบต้องฉลาดขึ้นทุกสัปดาห์
- หมายถึงมีการอัปเดต labels, retrieval memory, validation report,
  package registry, failure pattern, regime mapping อย่างต่อเนื่อง

8) DUPLICATE PREVENTION DESIGN
============================================================

เป้าหมาย:
- ไม่ให้ข้อมูลซ้ำเข้าฐานกลาง
- ไม่ให้ข้อมูลแท่งเวลาซ้ำหลัง resample
- ไม่ให้ weekly run สร้างผลซ้ำ
- ไม่ให้ bot ใช้ชุดข้อมูลผิดรุ่น

8.1 Duplicate types ที่ต้องกัน
D1. raw duplicate
- source row ซ้ำตรงตัว

D2. timestamp duplicate
- ts เดียวกันซ้ำหลายแถว

D3. semantic duplicate
- ts เดียวกัน แต่ค่า OHLCV ต่างกัน
- ต้องถือเป็น data conflict ไม่ใช่ duplicate ธรรมดา

D4. batch duplicate
- ingest file เดิมซ้ำ
- ต้อง detect จาก file hash

D5. run duplicate
- strategy run เดิมถูกบันทึกซ้ำ
- ต้องกันด้วย run fingerprint

D6. label duplicate
- case เดิมถูก label ซ้ำโดยไม่เปลี่ยน label_version

8.2 Duplicate prevention layers

Layer 1: file-level dedup
- คำนวณ SHA256 ของไฟล์ต้นทาง
- ถ้า file hash เดิมเคย ingest แล้ว ให้ mark เป็น duplicate_batch
- ไม่ append เข้าระบบซ้ำ

Layer 2: row-level dedup
- สร้าง row hash จาก symbol + ts + OHLCV
- ถ้า row hash ซ้ำ ให้ drop
- ถ้า ts ซ้ำแต่ค่าไม่เหมือนเดิม ให้ flag conflict

Layer 3: canonical uniqueness
- bars_canonical บังคับ unique (symbol, timeframe, ts)
- write ต้องใช้ merge/upsert logic

Layer 4: run fingerprint
- strategy_id + version + symbol + timeframe + window + param_hash + data_snapshot_id
- ถ้าซ้ำ ห้ามบันทึกผลเป็น run ใหม่

Layer 5: label versioning
- label = key(case_id, label_name, label_version)
- ถ้า case เดิม label_version เดิม ห้าม insert ซ้ำ

8.3 Conflict policy
กรณี ts ซ้ำแต่ OHLC ต่าง
- ห้ามเขียนทับเงียบ ๆ
- บันทึกลง conflict table
- mark quality_status = CONFLICT
- ต้องมี review log

8.4 Incremental append policy
- append เฉพาะช่วงเวลาใหม่
- ถ้าพบ overlap ต้อง compare hash ก่อน
- overlap เหมือนเดิม = skip
- overlap ต่าง = conflict review
