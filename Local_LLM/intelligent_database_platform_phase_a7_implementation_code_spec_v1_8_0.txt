1) LOCKED IMPLEMENTATION DECISIONS
============================================================

1. Primary contract gate
   - Pandera (Polars support)
   - ใช้สำหรับ schema validation, type checks, column checks, dataframe checks
   - ใช้เป็น PASS / WARN / FAIL / QUARANTINE decision gate ก่อนเข้า canonical layer

2. Critical-path processing engine
   - Polars LazyFrame-first
   - ใช้ lazy scan / transform / filter / dedup / aggregate / resample
   - งานที่ data-heavy ต้องพยายามอยู่ใน lazy pipeline ให้มากที่สุด

3. Canonical storage
   - Parquet partitioned dataset
   - ใช้ zstd compression เป็น default
   - เขียนแบบ deterministic path layout

4. Analytical / audit registry
   - DuckDB
   - ใช้เก็บ run registry, anomaly summary, quarantine index, strategy package registry,
     validation summary, publish status, service metadata

5. Similar-case memory / retrieval
   - Qdrant
   - ไม่อยู่ใน ingest hot path
   - อัปเดตหลัง labels/cases ถูก publish แล้ว

6. Backtest execution
   - Custom Numba-based simulation engine
   - ใช้ Ray สำหรับ batch fan-out เมื่อจำนวน scenario มาก

7. Service access model
   - Python SDK = primary local access
   - HTTP API = secondary multi-bot access
   - ทั้งสองทางต้องอิง registry/manifest เดียวกัน

3) MODULE-BY-MODULE RESPONSIBILITY
============================================================

3.1 contracts/pandera_ohlcv_schema.py
Purpose:
- นิยาม canonical OHLCV schema สำหรับ M1
- กำหนด field type, nullable rule, column checks, dataframe checks

Responsibilities:
- validate timezone-normalized timestamp
- validate numeric coercion
- validate low <= min(open, close, high)
- validate high >= max(open, close, low)
- validate volume >= 0
- validate no nulls in critical fields
- validate uniqueness of logical key ใน batch หลัง normalize

Outputs:
- validation result object
- failed rows dataframe
- severity classification

3.2 contracts/contract_thresholds.py
Purpose:
- เก็บ threshold policy ทั้งหมดแบบ central
- ใช้โดย normalize, anomaly scoring, validation, quarantine, publish quality gate

Responsibilities:
- spread-like bar range threshold
- zero-volume threshold policy
- gap threshold policy
- duplicate conflict tolerance
- rolling MAD / IQR multiplier
- warn/fail/quarantine cutoffs

Format:
- Python dict or dataclass constants
- ต้องมี version field
- เปลี่ยน threshold ต้อง bump contract version

3.3 ingest/mt5_pull_m1.py
Purpose:
- ดึง M1 จาก MT5 แบบ raw ingest source เดียว

Responsibilities:
- fetch by symbol/time window
- normalize column names from MT5
- stamp source metadata
- create raw batch id
- write raw parquet batch
- emit raw manifest

Rules:
- ห้าม resample ในโมดูลนี้
- ห้าม dedup ในโมดูลนี้
- ห้าม overwrite raw batch เดิม

3.4 ingest/raw_batch_manifest.py
Purpose:
- สร้าง metadata ของ raw batch ทุกก้อน

Manifest fields:
- run_id
- batch_id
- source_name
- source_symbol
- timeframe
- start_ts_utc
- end_ts_utc
- pulled_at_utc
- row_count
- file_path
- sha256_file
- schema_version
- source_session_id

3.5 ingest/normalize_ohlcv.py
Purpose:
- แปลง raw batch ให้เป็น canonical M1 frame

Responsibilities:
- rename columns -> canonical names
- cast numeric types
- cast timestamp to UTC
- sort by timestamp asc
- add symbol/timeframe/source/run_id
- derive logical_key
- derive row_hash
- attach batch_id
- strip impossible rows before contract validation only if clearly corrupt parse issue
- preserve rejected rows separately

3.6 ingest/continuity_checks.py
Purpose:
- ตรวจความต่อเนื่องของเวลาและจำแนก gap type

Responsibilities:
- missing minute detection
- duplicate timestamp detection
- reversed ordering detection
- abnormal gap clustering
- classify gap:
  - market_closed_expected
  - source_gap_suspect
  - partial_batch_overlap
  - timestamp_conflict
  - unknown_gap

Output:
- continuity summary
- row-level continuity flags
- quarantine recommendations

3.7 ingest/dedup_engine.py
Purpose:
- จัดการ row-level dedup และ conflict resolution แบบ deterministic

Decision order:
1. exact row_hash duplicate -> keep first canonical, drop duplicate
2. same logical_key + same OHLCV -> drop duplicate
3. same logical_key + different OHLCV -> mark conflict
4. choose winner only if trusted_source_priority rule exists
5. else send all conflicting candidates to quarantine and do not upsert

Rules:
- dedup engine ห้ามใช้อัลกอริทึม probabilistic
- ผลต้อง reproducible 100%
- ทุก decision ต้องถูก log เข้า DuckDB registry

3.8 ingest/quarantine_writer.py
Purpose:
- เก็บ rows/batches ที่ FAIL หรือ CONFLICT หรือ suspicious
- แยกตาม reason code และ run_id

Quarantine reasons:
- schema_fail
- timestamp_conflict
- duplicate_conflict
- continuity_gap_suspect
- ohlcv_impossible
- extreme_outlier
- parse_error
- source_inconsistency

Output layout:
- data/quarantine/reason_code=.../date=YYYY-MM-DD/*.parquet

3.9 ingest/canonical_writer.py
Purpose:
- เขียน canonical M1 dataset แบบ append-safe + idempotent publish rules

Rules:
- write only PASS or WARN rows that are not in conflict
- partition by symbol/year/month
- maintain manifest entry in registry
- publish only after write verification succeeds
- keep immutable published partitions
- corrections use replacement manifest, not silent overwrite

3.10 ingest/resample_timeframes.py
Purpose:
- สร้าง TF อื่นจาก canonical M1 แบบ deterministic

Mandatory TF:
- M2, M3, M4, M5, M6, M10, M15, M30, H1, H4, D1

Rules:
- open = first
- high = max
- low = min
- close = last
- tick_volume = sum
- source anchor = canonical M1 only
- resample output must carry parent_manifest_id and build_run_id

============================================================
5) QUALITY GATES
============================================================

Gate 1: Raw batch acceptance
- file readable
- manifest complete
- symbol/timeframe/source not null

Gate 2: Contract validation
- Pandera schema passes or warns within allowed threshold

Gate 3: Continuity and dedup
- no unresolved timestamp conflict enters canonical

Gate 4: Canonical publish verification
- row count + min/max ts + checksum summary logged

Gate 5: Feature build verification
- required features present for target timeframe

Gate 6: Research quality gate
- minimum trade count
- no broken validation windows
- no metric manifest mismatch

Gate 7: Publish quality gate
- approved package set exists
- intelligence snapshot generated
- publish registry committed

============================================================
6) FLEXIBLE ACCESS DESIGN FOR MANY FUTURE BOTS
============================================================

Principle:
- หนึ่ง platform กลาง
- หลาย bot เรียกใช้ได้
- ไม่บังคับให้ทุก bot ใช้ stack เดียวกัน

Access pattern:
A. Local Python bot
   -> ใช้ SDK โดยตรง
   -> latency ต่ำสุด
   -> เหมาะกับ MT5 execution bot

B. Local network bot / service bot
   -> ใช้ HTTP API
   -> รองรับ bot คนละ process/ภาษา
   -> เหมาะกับ dashboard bot / alert bot / LLM service bot

C. Batch consumer
   -> อ่าน publish parquet/json summaries
   -> เหมาะกับ research notebook / reporting layer

Rules:
- active published views only for bot consumption
- bot ห้ามอ่าน staging/quarantine/raw โดยตรง
- ทุก service response ต้องแนบ publish_version และ data_timestamp

============================================================
7) SECURITY / ROBUSTNESS RULES
============================================================

1. Raw data immutable
2. Canonical publish append-safe
3. Conflict rows never silently overwrite
4. Registry update inside transaction
5. Every run has run_id + manifest
6. Every publish has active_version + rollback pointer
7. Service routes are read-mostly
8. Live feedback writes are idempotent
9. Bot-facing data only comes from published views
10. Full lineage must trace:
    raw batch -> canonical manifest -> feature build -> backtest run -> package publish -> live feedback
