PURPOSE
- Define the production-grade technical specification for the standalone Intelligent
  Database Platform.
- This platform must serve multiple future bots with flexible access patterns.
- The platform must ingest, clean, deduplicate, resample, backtest, label, evaluate,
  index, publish, and serve intelligence as a central service.
- Bot Runtime Platform must consume this platform instead of rebuilding the same logic.

LOCKED PRINCIPLES
1. Database/Backtest is a standalone platform, not a bot-local utility.
2. Historical backtest outputs, labels, and case memory are central assets.
3. Every research strategy must be tested across all required timeframes:
   M1, M2, M3, M4, M5, M6, M10, M15, M30, H1, H4, D1.
4. Micro exit is mandatory as the base exit layer across every research timeframe.
5. Canonical source must be deterministic and deduplicated before any resampling.
6. Weekly intelligence refresh must be incremental by default and full rerun only when
   strategy logic or label logic changes materially.
7. Platform access must be flexible enough for multiple bot types in the future:
   signal bots, LLM confirm bots, monitor bots, portfolio bots, analytics bots,
   dashboard bots, and audit bots.

1) TARGET ARCHITECTURE
================================================================================

1.1 Platform split
A. Intelligent Database Platform
   - data ingest
   - canonical dataset creation
   - duplicate prevention
   - feature store
   - backtest engine
   - validation engine
   - label engine
   - retrieval memory engine
   - strategy package registry
   - intelligence publishing
   - service interfaces

B. Bot Runtime Platform
   - real-time data pull
   - candidate setup generation
   - service request to database platform
   - local LLM reasoning
   - execution confirmation
   - MT5 order routing
   - order monitoring
   - feedback write-back to database platform

1.2 Design objective
The database platform must become smarter every week.
Bots must become lighter over time because they consume higher-quality intelligence
from the central platform.

1.3 Core service model
The platform exposes:
- latest regime intelligence
- approved packages by timeframe
- approved micro exits by timeframe
- top-k similar historical cases
- live risk context
- recent failure patterns
- TP/SL surface recommendation
- execution constraints
- weekly intelligence snapshot
- audit-ready research evidence

2) MODERN TOOLCHAIN LOCK
================================================================================

2.1 Core stack
- Python = orchestration and system glue
- Polars = high-speed data processing
- Parquet = canonical storage and analytical persistence
- DuckDB = metadata and analytical query layer
- Qdrant = similarity retrieval and case memory index
- Custom Numba-based simulation engine = research and backtest core
- Ray = parallel scaling for TF x strategy x regime x walk-forward jobs

2.2 Why this stack
- Fast batch research
- Good local-machine performance
- Strong repeatability
- Easy versioned outputs
- Clean separation between tabular analytics and vector retrieval
- Flexible enough for future multi-bot ecosystem

2.3 Service access layer
Use two service styles:
- Internal Python SDK for local bots on the same machine
- HTTP API for remote or decoupled bots

Reason:
- SDK is fastest for same-host consumers.
- HTTP is more flexible for future distributed systems.
- Both must read from the same registry and response schema.

4) CANONICAL INGEST TECHNICAL SPEC
================================================================================

4.1 Canonical ingest objective
Convert all incoming market data into one trusted deterministic source.

4.2 Canonical M1 schema
Required fields:
- symbol: string
- broker_symbol: string
- source_type: string
- source_batch_id: string
- datetime_utc: timestamp
- open: float64
- high: float64
- low: float64
- close: float64
- volume: float64
- spread_points: float64 nullable
- tick_volume: float64 nullable
- ingest_ts_utc: timestamp
- data_hash: string
- row_status: string
- anomaly_flags: list/string

4.3 Canonical rules
- datetime must be UTC
- symbol must map to canonical symbol name (ex: GOLD)
- duplicate timestamps are not allowed after canonicalization
- open/high/low/close must be numeric and non-null
- high >= max(open, close, low)
- low <= min(open, close, high)
- rows must be sorted ascending by datetime_utc
- row_status must be ACTIVE, REJECTED, or QUARANTINED

4.4 Duplicate prevention standard
Use six layers:

Layer 1: raw batch fingerprint
- hash raw file payload or MT5 pull manifest
- prevent repeated import of same batch

Layer 2: row hash
- hash(symbol, datetime_utc, open, high, low, close, volume)
- detect exact duplicate rows

Layer 3: timestamp uniqueness
- only one ACTIVE row per symbol + timeframe + datetime_utc in canonical storage

Layer 4: conflict resolver
If same timestamp but different OHLC values:
- quarantine incoming row
- compare source_priority
- compare ingest recency
- compare external audit notes
- do not overwrite automatically unless replacement policy is triggered

Layer 5: continuity validator
- detect missing bars
- detect out-of-order rows
- detect abnormal gaps beyond expected market closure pattern

Layer 6: resample consistency validator
- verify derived TF bars exactly match canonical M1 aggregation

4.5 Recommended hash standard
- use SHA-256 for file/batch fingerprints
- use xxhash64 or blake3 for high-speed row hashing if speed is critical
- persist both logical primary key and row hash

4.6 Upsert policy
For canonical ACTIVE layer:
Primary logical key:
- symbol
- timeframe
- datetime_utc

Upsert rules:
- if new key: insert ACTIVE row
- if same key and exact same row hash: ignore
- if same key and different row hash: quarantine conflict and write to conflict log
- if manual or audited override approved: mark prior row as SUPERSEDED and promote new row

4.7 Quarantine policy
Conflict rows go to:
- data_quarantine/ohlcv_conflicts/
with:
- old_row_hash
- new_row_hash
- conflict_type
- source_priority
- review_status
- reviewer_note
- resolution_ts

5) TIMEFRAME BUILD SPEC
================================================================================

5.1 Mandatory research timeframes
- M1
- M2
- M3
- M4
- M5
- M6
- M10
- M15
- M30
- H1
- H4
- D1

5.2 Build rule
- Only M1 can be directly imported as canonical research source
- All other TFs must be derived from M1 by deterministic aggregation

5.3 Aggregation rule
Per TF candle:
- open = first open
- high = max high
- low = min low
- close = last close
- volume = sum volume

5.4 Rebuild policy
Incremental rebuild:
- rebuild only impacted windows after new M1 data arrives

Full rebuild:
- only when resample logic, timezone logic, or canonical conflict resolution changes

5.5 Continuity algorithm
For each symbol/timeframe:
- sort ascending
- compute expected delta by timeframe
- compare actual delta
- classify:
  - OK
  - EXPECTED_MARKET_CLOSED
  - MISSING_BAR
  - OVERLAP
  - OUT_OF_ORDER
- write continuity report to DuckDB and parquet artifact

9) MEMORY / RETRIEVAL SPEC
================================================================================

9.1 Objective
Provide top-k similar historical cases to bots and local LLM.

9.2 Why separate Qdrant from DuckDB
- DuckDB is best for analytical tables
- Qdrant is best for similarity search with metadata filters

9.3 Case memory unit
Each case should include:
- case_id
- symbol
- timeframe
- timestamp
- regime label
- feature snapshot
- strategy package id
- entry side
- entry context summary
- exit outcome summary
- MAE/MFE summary
- confidence label
- failure flags
- embedding vector
- metadata filters

9.4 Retrieval query modes
- by current timeframe only
- by regime + timeframe
- by session + trend direction
- by package id
- by failure pattern
- by live trade monitoring state

9.5 Top-k service rule
Default:
- fetch top 20 similar cases
- enforce metadata filters before similarity ranking when possible
- do not dump raw history into LLM
- send compressed evidence summary + selected representative cases

11) WEEKLY INTELLIGENCE RUN SPEC
================================================================================

Weekly run flow:

Step 1: pull new data from MT5
Step 2: fingerprint and validate raw batch
Step 3: canonicalize M1
Step 4: run duplicate and continuity checks
Step 5: rebuild impacted TF windows
Step 6: refresh feature store
Step 7: run backtest batches
Step 8: run walk-forward / validation batches
Step 9: refresh labels
Step 10: refresh memory cases
Step 11: refresh Qdrant index
Step 12: compare live results vs research expectations
Step 13: publish approved package updates
Step 14: publish weekly intelligence snapshot
Step 15: write progression update text
Step 16: archive manifests, logs, and run metadata

Default mode:
- incremental weekly run

Full research rerun triggers:
- entry logic changed
- exit logic changed
- feature logic changed
- label logic changed
- timeframe build logic changed
- severe regime drift detected
- canonical conflict resolution changed

14) ROBUSTNESS AND RELIABILITY SPEC
================================================================================

14.1 Reliability rules
- immutable raw layer
- deterministic canonical layer
- quarantine on conflict
- explicit versioning
- reproducible backtest runs
- registry-driven package promotion
- separation of analytics tables and vector memory
- weekly manifests and audit logs
- no silent overwrite

14.2 Performance rules
- Parquet partitioning by symbol/timeframe/date
- Polars lazy scans where possible
- DuckDB for aggregate queries and reporting
- Numba for repeated simulation loops
- Ray for parallel job execution
- incremental rebuild first

14.3 Recovery rules
- every weekly run has run_id
- every artifact stored under run_id
- failed stages must be restartable
- partial stage completion must not corrupt published registry
- publish step only after validation passes

19) FINAL LOCKED CONCLUSION
================================================================================

The correct long-term design is:

1. Intelligent Database Platform is the central brain for research memory,
   labels, package selection, and historical intelligence.
2. Bot Runtime Platform is a thin, fast execution client that requests guidance
   from the database platform and sends live outcomes back.
3. This split prevents repeated rework, supports many future bot types, improves
   maintainability, and allows weekly intelligence compounding.
