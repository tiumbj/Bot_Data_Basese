============================================================
1) CORE DECISION
============================================================

จากนี้ห้ามผสม 2 ส่วนนี้เข้าด้วยกัน

A. Intelligent Database Platform
หน้าที่:
- ingest historical + realtime market data
- build canonical dataset
- run backtest
- generate labels
- build memory cases
- update retrieval index
- publish weekly intelligence snapshot
- serve data/results to any bot

B. Bot Runtime Platform
หน้าที่:
- ขอข้อมูล / query จาก Intelligent Database Platform
- รับ signal context / regime / similar cases / approved packages
- สร้าง candidate setup
- ส่ง realtime multi-TF summary เข้า local LLM
- รับ final confirm / exit analysis
- ส่งคำสั่งเข้า MT5
- monitor order
- ส่งผล order กลับไปเก็บที่ Intelligent Database Platform

หลักการ:
Database Platform = สมองกลาง + ห้องวิจัย + knowledge service
Bot Platform = ผู้ใช้งานข้อมูลกลางเพื่อเทรดจริง

============================================================
2) WHY THIS SPLIT IS CORRECT
============================================================

เหตุผลที่ต้องแยก:
1. ลดการพังจากการแก้ bot แล้วกระทบ research layer
2. ทำให้ bot อื่นมาใช้ข้อมูลกลางได้ทันที
3. ทำให้ backtest/label/memory กลายเป็นสินทรัพย์กลางของทั้งระบบ
4. ทำให้รอบ weekly research และ live trading แยกกันชัด
5. ทำให้ versioning, validation, rollback, and audit ง่ายขึ้น
6. ทำให้ระบบ database platform สามารถพัฒนาให้ฉลาดขึ้นเรื่อย ๆ โดยไม่ต้องแก้ bot ทุกครั้ง

============================================================
4) PART A - DATA ARCHITECTURE
============================================================

4.1 Storage ชั้นหลัก
- Parquet = storage format หลัก
- DuckDB = analytical query engine หลัก
- Polars = data processing pipeline หลัก
- Qdrant = similar-case retrieval / vector search layer
- Python = orchestration layer

4.2 ทำไมเลือกชุดนี้
- Parquet อ่านเร็ว, บีบอัดดี, partition ได้
- DuckDB เหมาะกับ analytical query บนเครื่องเดียว
- Polars เร็วมากกับข้อมูลขนาดใหญ่
- Qdrant เหมาะกับ retrieval ของเคสเทรด
- Python เหมาะกับ orchestration และ integration กับ MT5

4.3 ห้ามใช้ CSV เป็น storage หลัก
CSV ใช้ได้เฉพาะ export/debug
ห้ามใช้เป็น source-of-truth

============================================================
5) PART A - DATABASE PLATFORM MODULES
============================================================

A1. data_ingest_mt5.py
หน้าที่:
- ดึง historical/recent bars จาก MT5
- append ข้อมูลใหม่
- กัน duplicate
- ตรวจ gap
- normalize timezone
- publish raw ingest log

A2. timeframe_builder.py
หน้าที่:
- สร้างทุก timeframe วิจัยจาก canonical source
- validate open/high/low/close alignment
- เก็บ bars แบบ partitioned

A3. feature_store_builder.py
หน้าที่:
- สร้าง trend, volatility, structure, session, liquidity, momentum features
- version feature set
- publish feature tables

A4. strategy_registry.py
หน้าที่:
- เก็บ strategy ids
- เก็บ strategy logic version
- เก็บ entry/exit package mapping
- เก็บ active/inactive state

A5. backtest_runner.py
หน้าที่:
- รัน full / rolling / sanity backtest
- วัด metrics
- export trade-level records

A6. label_engine.py
หน้าที่:
- triple-barrier labels
- meta-labeling
- custom failure labels
- custom early-exit labels
- outcome taxonomy versioning

A7. memory_case_builder.py
หน้าที่:
- แปลง trade outcomes เป็น memory cases
- รวม snapshot + label + summary + outcome
- เก็บ context ก่อนเข้า / ตอนถือ / ก่อน exit

A8. retrieval_index_builder.py
หน้าที่:
- encode memory cases
- update Qdrant
- validate index consistency
- update metadata filters

A9. weekly_intelligence_runner.py
หน้าที่:
- run weekly rolling research
- validate approved strategy packages
- detect drift
- detect regime transitions
- publish intelligence snapshot

A10. publish_service.py
หน้าที่:
- เปิดให้อ่านข้อมูลล่าสุด
- serve approved packages
- serve similar cases
- serve weekly winners
- serve market regime summaries
- serve exit failure patterns
