============================================================
0. OBJECTIVE
============================================================
เอกสารนี้เป็น Phase A9 ของ Intelligent Database Platform
หน้าที่ของเอกสารนี้คือ:
1. ล็อก registry schema กลางที่ใช้กับ production
2. ล็อกตาราง metadata / audit / package / publish / live feedback
3. ล็อก dependency map ระหว่าง ingest -> canonical -> features -> backtest -> labels -> publish -> bot access
4. ล็อก write contract แบบ idempotent เพื่อให้ bot หลายตัวใช้งานร่วมกันได้โดยไม่สะดุด
5. ล็อก SQL naming, index policy, event lineage, retention, and migration rules

ข้อสรุปสำคัญ:
- DuckDB = registry / analytical metadata plane
- Parquet = primary data lake storage
- Qdrant = similarity retrieval plane
- FastAPI + Python SDK = access plane
- Bot ห้าม query storage ภายในตรง
- Bot อ่านผ่าน service/SDK/API เท่านั้น

============================================================
1. ARCHITECTURE DECISION
============================================================
1.1 Source of truth
- Time-series canonical OHLCV = Parquet
- Feature store = Parquet
- Backtest outputs = Parquet + registry summary in DuckDB
- Labels / intelligence summaries / package registry / publish state / live feedback registry = DuckDB
- Similar case vectors = Qdrant

1.2 Read pattern
- Heavy analytical read = DuckDB scans Parquet/views
- Bot production read = SDK/API read from published tables/views only
- Retrieval read = Qdrant + published metadata join

1.3 Write pattern
- Raw ingest write -> staging parquet
- Canonical accepted write -> canonical parquet
- Quarantine write -> quarantine parquet + registry row in DuckDB
- Backtest result write -> result parquet + registry summary row
- Label write -> label parquet + registry summary row
- Publish write -> DuckDB publish registry + pointer to artifacts
- Live feedback write -> DuckDB event tables

============================================================
2. REGISTRY DESIGN PRINCIPLES
============================================================
2.1 Registry goals
- Traceability
- Idempotency
- Fast production lookup
- Safe multi-bot access
- Stable publish contract
- Easy rollback

2.2 Registry rules
- Every pipeline run must have run_id
- Every produced artifact must have artifact_id
- Every published package must have package_id and publish_version
- Every bot write must carry source_bot_id + event_id + event_ts_utc
- Every mutable business state must be reconstructable from append-only events or versioned tables

2.3 Naming rules
- snake_case only
- timestamps use *_ts_utc
- booleans use is_* / has_*
- primary keys end with _id
- foreign keys must use exact referenced key name
- views use vw_
- append-only logs use evt_
- dimension tables use dim_
- run tables use run_
- publish tables use pub_
- lookup tables use lk_

============================================================
3. CORE KEYS AND IDENTIFIERS
============================================================
3.1 Required IDs
- run_id: unique execution id for one orchestrated pipeline run
- artifact_id: unique id for one physical output artifact
- package_id: stable id for one strategy/micro-exit/research package
- publish_id: unique id for one publish action
- event_id: unique id for one incoming bot/system event
- trade_uid: unique trade lifecycle id
- case_id: unique similar-case memory id
- label_batch_id: unique label generation batch id
- feature_batch_id: unique feature generation batch id

3.2 ID format recommendation
- Prefix + UTC compact ts + short hash
Examples:
  run_20260320T093000Z_ab12cd34
  pkg_m30_pullbackdeep_2f93e1a1
  evt_exec_20260320T094002Z_55ab8891

3.3 Idempotency keys
- ingest batch: source_name + symbol + timeframe + source_window_start + source_window_end + raw_batch_hash
- bot event: source_bot_id + event_type + external_ref + event_ts_utc bucket + payload_hash
- publish: package_id + publish_scope + target_alias + content_hash

============================================================
9. SQL CONTRACT RULES
============================================================
9.1 No direct bot access
- Bots must not read internal schemas except through service layer
- Allowed bot-facing reads are materialized read models / API payloads / SDK responses only

9.2 Append-only events
- live.trade_event_log is append-only
- Corrections must be new events, not in-place edits

9.3 Idempotent writes
- All write endpoints must require idempotency_key
- Duplicate keys return accepted previous result, not error cascade

9.4 Rebuildable projections
- trade_state_projection may be dropped and rebuilt from event log
- active_alias_registry may be rebuilt from publish_registry

9.5 Immutable artifacts
- Once artifact content_hash is registered, artifact content must not mutate
- New content = new artifact_id

============================================================
12. ROLLBACK AND FAILOVER RULES
============================================================
12.1 Publish rollback
- Never overwrite active package content in place
- Create new publish row with rollback reference
- Update active_alias_registry atomically

12.2 Event ingest failover
- If live write fails transiently, queue locally and retry with same idempotency_key
- If retry exhausted, write dead-letter record

12.3 Read failover
- If new publish artifact unavailable, bots continue reading previous active alias
- Fail closed for write corruption, fail open for read-only stale package if freshness within allowed budget

============================================================
13. RETENTION POLICY
============================================================
- run_registry: permanent
- artifact_registry: permanent metadata; artifact retention depends on class
- quarantine_registry: permanent metadata, payload artifacts rolling >= 180 days recommended
- trade_event_log: permanent or archive-only append strategy
- trade_state_projection: rebuildable, no hard permanence required
- publish_registry: permanent
- active_alias_registry: current state only, rebuildable

============================================================
14. MIGRATION POLICY
============================================================
- Every schema change must increment schema_version
- Backward-compatible additions first
- Do not rename/remove bot-facing fields without new read_model_version
- Migration scripts must be additive where possible
- Breaking changes require dual-publish window
