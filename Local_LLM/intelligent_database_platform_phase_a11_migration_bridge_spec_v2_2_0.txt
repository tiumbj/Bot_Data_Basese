============================================================
1. PURPOSE
============================================================

เอกสารนี้กำหนดวิธีเชื่อมผลการทดสอบที่มีอยู่แล้วในโปรเจกต์ปัจจุบัน
เช่น build outputs, package outputs, validation results, timeframe results, micro-exit results,
benchmark outputs, summary logs, และ strategy notes
ให้เข้าสู่ Intelligent Database Platform แบบไม่ต้องรื้อ pipeline เดิม

หลักการสำคัญ:
- current tests continue
- current outputs are research assets
- new platform absorbs and standardizes them
- migration must be additive only
- no disruption to current research momentum
- no rewrite of existing build/test jobs required in phase A11

LOCKED PRINCIPLE:
Current tests continue; future platform absorbs and standardizes them, not replaces them.

============================================================
2. A11 OBJECTIVE
============================================================

A11 มีเป้าหมาย 6 ข้อ:

1) รับผลลัพธ์จากงาน test/backtest ที่มีอยู่แล้วเข้า schema กลาง
2) ทำให้ผลย้อนหลังกลายเป็น historical research asset อย่างเป็นระบบ
3) ไม่แก้ logic หลักของ pipeline เดิม
4) เพิ่ม traceability จาก artifact -> registry -> package candidate
5) ทำให้ bot ในอนาคตเรียกใช้ผลวิจัยเดิมได้
6) ปูทางสู่ weekly automatic ingestion ของผลวิจัยใหม่

============================================================
3. WHAT CURRENTLY EXISTS AND MUST BE PRESERVED
============================================================

สิ่งที่ถือว่าเป็น existing assets และห้ามทิ้ง:

A. strategy build outputs
B. entry component package outputs
C. micro exit research outputs
D. per-timeframe test outputs
E. validation outputs
F. benchmark outputs
G. walk-forward or batch summary outputs
H. logs ที่บอก parameter / timeframe / metrics / package name
I. manual research notes หรือ summary text
J. generated datasets/index files ที่ใช้เป็นหลักฐานการรัน

ของพวกนี้จะถูกจัดชั้นเป็น:
- raw research artifact
- normalized research record
- package candidate
- active published package (เมื่อผ่านเกณฑ์)

============================================================
4. MIGRATION PHILOSOPHY
============================================================

A11 ใช้หลัก 4 ชั้น:

LAYER 1: Preserve
เก็บ output เดิมไว้ตามเดิม ห้ามแก้ย้อนหลังโดยไม่จำเป็น

LAYER 2: Discover
ค้นหาและทำ inventory ว่ามีไฟล์อะไรบ้าง อยู่ที่ไหน สร้างจาก job ไหน

LAYER 3: Normalize
อ่าน output เดิม แล้ว map เข้า canonical registry schema กลาง

LAYER 4: Publish
สร้าง package candidate / evidence record / intelligence summary จากข้อมูลที่ normalize แล้ว

A11 ไม่ทำ:
- rewrite build pipeline เดิม
- เปลี่ยนชื่อไฟล์เดิมทั้งหมด
- บังคับ schema เดิมให้เหมือนใหม่ในทันที
- ลบผลการทดสอบเก่า

============================================================
5. HIGH-LEVEL MIGRATION ARCHITECTURE
============================================================

CURRENT RESEARCH JOBS
    -> produce raw files / logs / summaries / package outputs
    -> bridge scanners discover artifacts
    -> bridge parsers extract metadata + metrics + lineage
    -> bridge normalizers map into central schema
    -> registry writers write append-only records
    -> package candidate evaluator marks usable assets
    -> publish layer exposes approved assets to bots

สรุป:
เดิมยังรันเหมือนเดิม
ใหม่เพิ่ม "bridge layer" มารับของเดิมเข้า platform กลาง

7. research_note_record
เก็บข้อความสรุป, ข้อสังเกต, edge case, caveat

8. research_lineage_record
เก็บความสัมพันธ์ระหว่าง source file -> parsed record -> package candidate

============================================================
9. MIGRATION CLASSIFICATION RULES
============================================================

ทุก artifact ต้องถูกจัดเข้าหนึ่งใน 5 คลาส:

CLASS A: fully structured
เช่น csv/json/parquet ที่ parse field ได้ชัด

CLASS B: semi-structured
เช่น txt/log ที่มี pattern parse ได้

CLASS C: weakly structured
เช่น summary text ที่ parse ได้บางส่วน

CLASS D: opaque evidence
เช่น binary output หรือ artifact ที่ยัง parse ไม่ได้ แต่เก็บเป็น evidence

CLASS E: rejected
ไฟล์เสีย / อ่านไม่ได้ / ซ้ำ / ขาดข้อมูลหลักเกิน threshold

แนวปฏิบัติ:
- A, B, C -> เข้าสู่ normalization
- D -> เก็บเป็น evidence only
- E -> quarantine registry

============================================================
10. MIGRATION BRIDGE MODULES
============================================================

A11 แนะนำให้มีโมดูลจริงดังนี้:

1) bridge_inventory_scanner.py
หน้าที่:
- scan โฟลเดอร์ผลวิจัย
- ทำ file inventory
- จัด class เบื้องต้น
- คำนวณ sha256
- บันทึก discovered artifacts

2) bridge_pattern_registry.py
หน้าที่:
- เก็บ regex / parsing rules / filename rules
- map ชื่อ package/strategy/timeframe จาก path และ text

3) bridge_artifact_parser.py
หน้าที่:
- parse structured/semi-structured artifacts
- คืน normalized intermediate dict

4) bridge_metric_normalizer.py
หน้าที่:
- map metric alias ให้เป็นชื่อมาตรฐาน
- normalize units / percentage / currency / points

5) bridge_strategy_resolver.py
หน้าที่:
- resolve strategy family / entry / exit / filter stack
- สร้าง strategy fingerprint

6) bridge_lineage_builder.py
หน้าที่:
- สร้าง relation source -> run -> metrics -> package candidate

7) bridge_registry_writer.py
หน้าที่:
- เขียนเข้า DuckDB registry แบบ append-only + idempotent

8) bridge_publish_evaluator.py
หน้าที่:
- จัดสถานะ candidate / baseline / rejected / publishable

9) bridge_notes_ingestor.py
หน้าที่:
- รับ txt notes / summary reasoning / conclusions เข้า note registry

10) bridge_migration_report.py
หน้าที่:
- สรุปว่ารับอะไรเข้าได้เท่าไร
- ขาดอะไร
- conflict อะไร
- coverage เป็นอย่างไร

============================================================
12. IDENTITY AND IDEMPOTENCY
============================================================

Bridge ต้อง idempotent

PRIMARY RULE:
artifact เดิมต้องไม่ถูก ingest ซ้ำเป็น record ใหม่ ถ้าเนื้อหาไฟล์เหมือนเดิม

ใช้ key ดังนี้:
artifact_identity_key =
    sha256(source bytes)
    + source_file_name
    + source_size_bytes

normalized_record_key =
    source_sha256
    + resolved_strategy_fingerprint
    + timeframe
    + dataset_id
    + parameter_signature

ถ้าเจอซ้ำ:
- ไม่เขียนซ้ำ
- อัปเดต last_seen_utc
- เพิ่ม reference count ได้
- เก็บใน lineage ว่าถูกพบอีกครั้งจาก scan ไหน

============================================================
13. METRIC NORMALIZATION STANDARD
============================================================

bridge_metric_normalizer ต้อง map alias ให้เหมือนกัน เช่น:

winrate, win_rate, pct_win -> win_rate
pf, profit_factor -> profit_factor
dd, drawdown, max_dd -> max_drawdown
net, net_pnl, total_profit -> net_profit
trades, n_trades, total_trades -> total_trades
rr, avg_rr, mean_rr -> avg_rr

หน่วย:
- เปอร์เซ็นต์ normalize เป็น 0-100 scale เสมอ
- เงิน normalize ตาม base currency หรือระบุ unknown
- points / pips ต้องมี field unit แยก
- ห้ามรวมคนละหน่วยใน field เดียว

19. MIGRATION QUALITY LEVELS
============================================================

ทุก bridge batch ต้องให้คะแนนคุณภาพ:

LEVEL 1
เก็บได้แค่ artifact inventory

LEVEL 2
parse metadata ได้บางส่วน

LEVEL 3
parse metrics มาตรฐานได้

LEVEL 4
สร้าง strategy fingerprint + lineage ได้

LEVEL 5
พร้อม promote เป็น candidate/publishable

เป้าของ A11 คือให้ผลเดิมจำนวนมากที่สุดขึ้นสู่ LEVEL 3-4 ก่อน
ไม่จำเป็นต้องบังคับทุกอย่างเป็น LEVEL 5 ทันที
