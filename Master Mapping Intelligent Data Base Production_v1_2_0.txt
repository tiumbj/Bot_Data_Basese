[LOCKED ROOT CAUSE]
RC-EXEC-RESUME-001: Execution runner lacks resume-safe state and deterministic restart recovery.

[ROOT CAUSE DESCRIPTION]
Current execution runner can continue past per-job errors via continue-on-error, but it cannot safely resume after process kill, machine shutdown, restart, or interrupted shard execution. This causes:
1. repeated shard reruns,
2. output fragmentation across multiple outdirs,
3. wasted compute time,
4. higher operational risk for long-running research/backtest jobs,
5. non-production-safe execution recovery behavior.

[LOCKED PRODUCTION REQUIREMENT]
Any runner promoted to production execution standard must provide true resume-safe execution.

[LOCKED ROOT CAUSE]
RC-GOV-EXEC-003: Known execution defects were identified in the current runner, but the same runner continued to be used as active execution infrastructure. This created a governance failure: audit findings were not converted into blocking execution rules.

[DESCRIPTION]
The project already proved that v1.0.8 has major execution weaknesses:
- no resume-safe state,
- partial shard stalls,
- fragmented rerun workflow,
- high orchestration / I/O overhead.

Despite that, the runner continued to be used for active execution. This means the problem is not only technical but also operational-governance related.

[LOCKED RULE]
Any runner with known blocking execution defects must be downgraded immediately to:
- research-use only, or
- controlled seed execution only.

It must not continue as the default execution standard.

[MANDATORY ACTION]
Audit findings must be converted into execution gates.
If a runner fails a production execution gate, future runs on that runner must be explicitly marked as temporary/non-production.

[MANDATORY CAPABILITIES]
1. Persistent execution state store
   - Must store shard state, group state, and job state in a persistent store.
   - SQLite is acceptable as minimum production baseline.

2. Deterministic restart recovery
   - Restarting the same shard in the same outdir must resume from prior state.
   - Completed jobs/groups must be skipped automatically.
   - No manual outdir fork should be required for normal recovery.

3. Checkpoint granularity
   - Must checkpoint at least at group-level.
   - Preferred: group-level + job-level status tracking.

4. Progress reconstruction
   - live progress must be reconstructable from persisted state after restart.
   - Progress numbers must not reset to zero unless state is intentionally reset.

5. Output consistency
   - Result outputs must remain deterministic under restart.
   - Restart must not duplicate already completed result rows.
   - Restart must not require manual merge of partially completed outputs under normal operation.

6. Recovery observability
   - Runner must report:
     - resumed_groups
     - resumed_jobs
     - skipped_completed_groups
     - skipped_completed_jobs
     - recovery_start_time
     - recovery_status

[LOCKED OPERATIONAL RULE]
A runner without resume-safe state may be used only for controlled seed execution or temporary research use.
It must not be treated as production-grade execution infrastructure.

[VERSION DECISION]
- v1.0.8 execution runner:
  allowed_for = controlled_seed_execution_only
  production_execution_standard = FAIL

- next promoted execution runner:
  must_pass = resume_safe_execution_requirement

[ACCEPTANCE TEST]
The runner is not considered production-safe until it passes all of the following:
1. Start shard execution.
2. Let it complete a measurable portion of groups/jobs.
3. Kill the process.
4. Restart with the same outdir and same shard arguments.
5. Verify:
   - completed work is skipped,
   - progress continues from prior persisted state,
   - results are not duplicated,
   - no manual merge is required,
   - recovery metrics are reported correctly.

[LOCKED DIRECTION]
Resume-safe recovery is mandatory for the next execution runner promotion.
No future claim of “production execution ready” is valid without this capability.
